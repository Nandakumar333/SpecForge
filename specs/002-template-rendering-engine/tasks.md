## Implementation Strategy

### MVP First (User Story 1 Only)

1. Complete Phase 1: Setup (data models + config)
2. Complete Phase 2: Foundational (template files + validator)
3. Complete Phase 3: User Story 1 (renderer + registry core)
4. **STOP and VALIDATE**: All built-in templates render correctly with validated context
5. Existing `specforge init` still works via backward-compat wrapper

### Incremental Delivery

1. Setup + Foundational → Foundation ready
2. Add US1 → Core rendering works → **MVP! Can generate all files**
3. Add US3 → Stack-specific prompts → .NET ≠ Node.js ≠ Python
4. Add US2 → User overrides → Teams can customize
5. Add US5 → Output validation → Quality gate on all output
6. Add US4 → Full catalog → Registry is queryable
7. Polish → Migrate scaffold pipeline → Remove deprecated code

### Key Architectural Notes

- **New modules built alongside old** — existing `template_loader.py` untouched until Phase 8
- **All new code uses Result[T, str]** — no exceptions for control flow
- **Constructor injection everywhere** — TemplateRenderer(registry), no globals
- **Custom filters are pure functions** — trivially testable, no side effects
- **TemplateValidator is stateless** — no constructor args, no shared state
- **Dot-notation for variants** — `backend.dotnet.md.j2` not `backend/dotnet.md.j2` (R-03)
