# Feature Specification: Implementation Orchestrator

**Feature Branch**: `011-implementation-orchestrator`  
**Created**: 2026-03-17  
**Status**: Draft  
**Input**: User description: "Build the orchestrator that manages multi-service implementation in correct order"

## Clarifications

### Session 2026-03-17

- Q: Should the orchestrator support partial --all (e.g., "implement --all --to-phase 2")? → A: Yes — support `--to-phase N` to allow incremental execution up to a specified phase.
- Q: How should it handle a service failing in the middle of a phase? → A: Complete remaining independent services in the current phase, then halt before proceeding to the next phase.
- Q: Should cross-service integration tests be auto-generated from contracts? → A: Yes — auto-generate integration tests from published contracts to ensure 100% contract coverage.
- Q: For docker-compose integration, should it use Testcontainers or actual Docker? → A: Actual docker-compose — already generated by shared-infra pre-phase, matches production deployment, no language-specific test dependency.
- Q: How should the orchestrator handle database migrations across services? → A: Database-per-service in microservice mode; schema-per-module with single database in monolith mode.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Phased Multi-Service Implementation (Priority: P1)

As a developer working on a microservice project, I want `specforge implement --all` to build every service in the correct dependency order with integration checks between phases, so that each layer of services is verified before dependent services are built on top of it.

The orchestrator reads the project manifest to determine architecture mode and service dependency graph, then groups services into execution phases. Services within the same phase have no mutual dependencies and can be implemented in parallel. Between each phase, the orchestrator runs contract verification and infrastructure health checks. Only when all verifications pass does the next phase begin.

**Why this priority**: This is the core value proposition — without phased orchestration, developers must manually sequence service builds, track dependencies, and remember to verify contracts between services. This story delivers the fundamental end-to-end automation that makes multi-service projects manageable.

**Independent Test**: Can be fully tested by running `specforge implement --all` on a project with 3+ services across 2+ dependency phases. Delivers value by automating the entire build sequence with zero manual intervention between phases.

**Acceptance Scenarios**:

1. **Given** a microservice project with 8 services declared in the manifest, each with explicit dependencies, **When** the developer runs `specforge implement --all`, **Then** the system builds shared infrastructure first, then implements services in dependency-ordered phases (foundation services with no dependencies first, then services that depend on those, and so on), running contract verification between each phase.
2. **Given** a microservice project where Phase 1 contains two independent services (identity-service and admin-service), **When** Phase 1 begins, **Then** both services are implemented (sequentially or in parallel as configured) and both must complete successfully before Phase 2 starts.
3. **Given** a microservice project where Phase 2 services (ledger-service, portfolio-service, integration-service) all depend on Phase 1's identity-service, **When** Phase 1 completes and passes verification, **Then** Phase 2 begins and all three services are implemented, each receiving identity-service's published contracts as read-only context.
4. **Given** a service in Phase 2 fails during implementation (build error, test failure beyond retry budget), **When** the failure is detected, **Then** the orchestrator allows remaining independent services in the same phase to complete (since they share no mutual dependencies), reports the failure with diagnostic context, and does not proceed to Phase 3.
5. **Given** a microservice project with 3 phases, **When** the developer runs `specforge implement --all --to-phase 2`, **Then** only shared infrastructure, Phase 1, and Phase 2 are executed (with inter-phase verification after each), and Phase 3 is not attempted regardless of Phase 2's outcome.

### User Story 2 - Inter-Phase Contract Verification (Priority: P1)

As a developer, I want the orchestrator to detect when a newly implemented service breaks an existing service's contract before proceeding to the next phase, so that integration problems are caught early rather than discovered after the entire project is built.

After each phase completes, the orchestrator runs contract verification between all implemented services (both from the just-completed phase and all prior phases). This includes checking that published API contracts match consumer expectations, event schemas are compatible, and shared entity definitions are consistent. If any contract violation is detected, the orchestrator stops, reports the specific mismatch, and waits for resolution before continuing.

**Why this priority**: Contract violations between services are the most expensive bugs in microservice projects. Detecting them immediately after the offending service is built (rather than at the end) saves hours of debugging. This is co-equal with phased execution because phased execution without verification is just sequential building with extra steps.

**Independent Test**: Can be tested by implementing two services where the second service produces outputs that conflict with the first service's expected inputs. The orchestrator must detect and report the mismatch at the phase boundary.

**Acceptance Scenarios**:

1. **Given** Phase 1 (identity-service) is complete and publishes a JWT claims contract, **When** Phase 2's ledger-service is implemented and changes the expected JWT claims format, **Then** the post-phase contract verification detects the mismatch between ledger-service's expectations and identity-service's published contract, and the orchestrator stops before proceeding to Phase 3.
2. **Given** Phase 2 is complete with three services, **When** post-phase verification runs, **Then** it checks contracts between all implemented services (Phase 1 + Phase 2), not just within Phase 2.
3. **Given** a contract violation is detected between two services, **When** the violation report is presented, **Then** it identifies the specific services, the conflicting contract elements, and which service introduced the breaking change.
4. **Given** the developer fixes the contract violation and re-runs the orchestrator, **When** it resumes, **Then** it re-verifies the previously failing contracts and proceeds to the next phase only if all now pass.
5. **Given** Phase 2 completes and all services in Phase 1 + Phase 2 pass contract verification, **When** the orchestrator proceeds, **Then** shared entity boundary analysis (from Feature 006) is also run to detect any entity ownership conflicts across the implemented services.

### User Story 3 - Full Integration Validation (Priority: P2)

As a developer, I want a comprehensive integration test after all services are implemented to confirm the entire system works end-to-end, so that I have confidence the project is deployment-ready.

After the final implementation phase completes and passes contract verification, the orchestrator runs a post-phase integration validation. This spins up all services together (including gateway, databases, and message broker), verifies health checks, tests a representative request flow through the system, validates event propagation between services, and produces an integration report summarizing the results.

**Why this priority**: While inter-phase verification catches contract mismatches, only a full integration test confirms that all services cooperate in a live environment. This is P2 because it depends on the phased execution and contract verification being in place first — it's the capstone validation, not the core orchestration.

**Independent Test**: Can be tested by running the post-phase integration suite on a fully implemented project. Delivers value by providing a single pass/fail verdict for the complete system.

**Acceptance Scenarios**:

1. **Given** all 8 services have been implemented and all inter-phase verifications passed, **When** the post-phase integration validation runs, **Then** all services plus the gateway, database, and message broker start up together and every service's health check passes within a configurable timeout.
2. **Given** all services are running, **When** a sample request is sent through the gateway, **Then** the request flows through the gateway to the appropriate service, the service processes it (including any downstream service calls), and a valid response is returned through the gateway.
3. **Given** all services are running, **When** an event-producing action occurs in one service, **Then** the event flows through the message broker and is received by all subscribing services.
4. **Given** the integration validation completes (pass or fail), **When** the report is generated, **Then** it includes per-service health status, gateway route verification results, cross-service request flow results, event propagation results, and a summary verdict.
5. **Given** one service fails its health check during integration validation, **When** the failure is detected, **Then** the report clearly identifies which service failed, includes its logs, and marks the overall integration as failed.

### User Story 4 - Simplified Monolith Orchestration (Priority: P2)

As a developer working on a monolithic project, I want `specforge implement --all` to implement modules in dependency order without Docker or contract tests, so that the orchestration adapts to my architecture without unnecessary complexity.

When the project manifest declares a monolithic (or modular-monolith) architecture, the orchestrator simplifies its execution model. Instead of phased execution with container orchestration, it implements modules in topological dependency order. Quality checks run after each module (build, lint, test) but skip container, contract, and inter-service checks. The final integration test runs the monolith as a single application with all modules enabled rather than spinning up multiple containers.

**Why this priority**: Monolith mode is essential for the tool to be useful across architecture types. It's P2 because the microservice orchestration is the harder, more differentiated problem — monolith orchestration is a simplified subset that reuses most of the same dependency resolution and sequencing logic.

**Independent Test**: Can be tested by running `specforge implement --all` on a monolithic project with 4+ modules. Delivers value by automating module-level build sequencing without microservice overhead.

**Acceptance Scenarios**:

1. **Given** a monolithic project with modules declared in dependency order (e.g., auth-module depends on nothing, billing-module depends on auth-module), **When** the developer runs `specforge implement --all`, **Then** modules are implemented in topological order (auth-module first, then billing-module) without any Docker, contract verification, or container health checks.
2. **Given** a monolithic project, **When** each module is implemented, **Then** standard quality checks run (build, lint, test) but microservice-specific checks (container build, contract tests, health endpoint) are skipped entirely.
3. **Given** all modules are implemented in a monolithic project, **When** the final integration test runs, **Then** it starts the monolith as a single application with all modules enabled and runs an end-to-end test suite against it.
4. **Given** a modular monolith project, **When** inter-module verification runs between phases, **Then** it checks module boundary compliance (no cross-module direct data access) instead of contract verification.

### User Story 5 - Shared Infrastructure Pre-Phase (Priority: P2)

As a developer in a microservice project, I want shared infrastructure (contracts library, docker-compose base, API gateway skeleton) to be built before any service implementation begins, so that every service has a consistent foundation to build upon.

Before any service phase starts, the orchestrator runs `specforge implement --shared-infra` (delegating to the existing shared infrastructure executor from Feature 009). This builds the contracts library, base docker-compose configuration, API gateway skeleton, message broker configuration, and any shared utilities. This pre-phase must complete successfully before any service implementation begins. If the shared infrastructure build fails, no service phases are attempted.

**Why this priority**: Shared infrastructure is a prerequisite for all microservice implementation but is already partially implemented in Feature 009's US4. This story is about the orchestrator correctly sequencing it as a mandatory pre-phase, not re-implementing the infrastructure build itself.

**Acceptance Scenarios**:

1. **Given** a microservice project, **When** `specforge implement --all` starts, **Then** shared infrastructure is built first before any service phase begins.
2. **Given** shared infrastructure build fails (e.g., contracts library has compilation errors), **When** the failure is detected, **Then** no service phases are attempted, and the orchestrator reports the failure with instructions to fix shared infrastructure first.
3. **Given** a monolithic project, **When** `specforge implement --all` starts, **Then** the shared infrastructure pre-phase is skipped entirely (no Docker, gateway, or broker configuration needed).

### User Story 6 - Orchestration State and Resumption (Priority: P3)

As a developer whose orchestration run was interrupted (manual cancellation, system crash, or a phase failure they've since fixed), I want to resume from where the orchestrator left off rather than re-implementing already-completed services.

The orchestrator persists its progress after each significant milestone (phase completion, verification pass, service completion). When resumed, it reads the saved state, identifies which phases and services are already complete, and continues from the next incomplete step. Already-completed services are not re-implemented, but their contracts are re-loaded for verification against new services.

**Why this priority**: Resumption is a quality-of-life feature that becomes critical for large projects where a full re-run may take significant time. It's P3 because the core orchestration must work correctly first — resumption is an optimization on top of a working system.

**Acceptance Scenarios**:

1. **Given** the orchestrator completed Phase 1 and Phase 2 but was interrupted during Phase 3, **When** the developer runs `specforge implement --all --resume`, **Then** Phases 1 and 2 are skipped (not re-run), and Phase 3 resumes from the first incomplete service.
2. **Given** the orchestrator was interrupted mid-service (e.g., ledger-service had 5 of 10 tasks complete), **When** resumed, **Then** the orchestrator delegates to the sub-agent executor's own resumption mechanism (Feature 009) for that service, and continues with the remaining services after it completes.
3. **Given** a previous run completed all phases but failed during post-phase integration validation, **When** resumed, **Then** the orchestrator skips all implementation phases and re-runs only the integration validation.
4. **Given** no previous orchestration state exists, **When** the developer runs `specforge implement --all --resume`, **Then** the system informs the user that no prior state was found and starts a fresh run.

---

### User Story 7 - Progress Visibility and Reporting (Priority: P3)

As a developer running a long multi-service build, I want clear real-time progress visibility showing which phase is active, which services are complete, and what's coming next, so that I can understand the orchestration status at a glance.

The orchestrator provides a structured progress display throughout execution. It shows the overall phase map (with completed/active/pending indicators), per-service status within the current phase, verification results between phases, and a final summary report upon completion (or failure). The display uses the project's existing terminal output system for consistent formatting.

**Why this priority**: Progress visibility is important for developer experience but doesn't affect the correctness of orchestration. It's P3 because a working orchestrator with minimal output is more valuable than a beautifully formatted one that doesn't orchestrate correctly.

**Acceptance Scenarios**:

1. **Given** the orchestrator is running, **When** a phase transition occurs (e.g., Phase 1 → verification → Phase 2), **Then** the display clearly shows the transition, including which services completed and the verification result.
2. **Given** the orchestrator completes (success or failure), **When** the final report is displayed, **Then** it includes: total services implemented, per-phase breakdown, verification results at each boundary, integration validation results (if reached), total elapsed time, and any services that were skipped or failed.
3. **Given** a service within a phase fails, **When** the failure is reported, **Then** the display shows which service failed, which services in the same phase succeeded, and what the consequences are (phase halted, no further phases attempted).

### Edge Cases

- What happens when the dependency graph has a cycle (Service A depends on B, B depends on A)? The orchestrator must detect the cycle before execution begins and report it as a configuration error with the specific services involved.
- What happens when a service has no spec artifacts (no spec.md, no tasks.md)? The orchestrator must skip that service, log a warning, and continue with services that have complete artifacts. If a skipped service is a dependency of another service, the dependent service is also skipped with a clear explanation.
- What happens when the manifest declares services that don't exist in the project directory? The orchestrator must report missing services before execution begins and refuse to start until the manifest matches the project structure.
- What happens when all services in a phase fail? The orchestrator reports all failures (since it allows independent services to continue within a phase) and does not attempt subsequent phases. The integration report details each failure independently.
- What happens when contract verification detects multiple violations across different service pairs? All violations are reported together (not just the first one) so the developer can address them in batch.
- What happens when the developer runs `specforge implement --all` while another orchestration is already running? The orchestrator must detect the concurrent run (via lock file) and refuse to start, informing the developer of the existing run.
- What happens when the project has only one service and no dependencies? The orchestrator runs the single service through the sub-agent executor with standard quality checks, effectively behaving as a pass-through with shared infrastructure pre-phase (if microservice) and post-phase integration validation.
- What happens when a service succeeds but its post-phase verification reveals it broke a previously passing contract? The orchestrator must report which service and which contract, and halt before the next phase. The offending service's implementation is not rolled back — the developer fixes forward.
- What happens when a service's database migration conflicts with another service's schema (e.g., both define a `users` table)? In microservice mode this cannot happen (database-per-service). In monolith mode, the module boundary compliance check detects cross-module schema ownership violations and reports them at the phase boundary.
- What happens when `--to-phase N` is specified but N exceeds the total number of phases? The orchestrator executes all phases (equivalent to omitting the flag) and logs a warning that the specified phase number exceeds the plan.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST read the project manifest to determine architecture mode (microservice, monolithic, modular monolith) and adapt orchestration behavior accordingly.
- **FR-002**: System MUST build a dependency graph from the manifest's declared service/module dependencies and compute a topologically sorted execution plan grouped into phases, where each phase contains services/modules with no mutual dependencies.
- **FR-003**: System MUST detect dependency cycles before execution begins and report them as configuration errors with the specific services/modules involved in the cycle.
- **FR-004**: System MUST execute shared infrastructure (contracts library, docker-compose base, API gateway skeleton, message broker configuration) as a mandatory pre-phase before any service implementation in microservice mode.
- **FR-005**: System MUST skip the shared infrastructure pre-phase entirely in monolithic mode.
- **FR-006**: System MUST implement services within each phase by delegating to the sub-agent executor (Feature 009) for each service, passing the service's spec artifacts and dependency contracts as context.
- **FR-007**: System MUST ensure context isolation between services — no service's implementation code is exposed to another service's sub-agent executor. Only published contracts (API specs, event schemas) from dependency services are shared.
- **FR-008**: System MUST run inter-phase verification after each phase completes, including: contract verification between all implemented services (current phase + all prior phases) and shared entity boundary analysis (Feature 006). Inter-phase verification is lightweight (contract file comparison + boundary analysis only, no Docker). Full infrastructure health checks (docker-compose up, health endpoints, request flow) run only in the post-phase final integration validation (FR-011).
- **FR-009**: System MUST halt execution and report failures if any inter-phase verification check fails, without proceeding to the next phase.
- **FR-010**: System MUST run module boundary compliance checks (no cross-module direct data access) instead of contract verification in modular monolith mode.
- **FR-011**: System MUST run a full integration validation after the final phase completes in microservice mode, including: starting all services together, health check verification for every service, gateway route verification, cross-service request flow testing, and event bus message propagation testing.
- **FR-012**: System MUST run a monolith integration test (single application with all modules enabled) after the final phase in monolithic mode.
- **FR-013**: System MUST persist orchestration state after each phase completion, service completion, and verification pass, enabling resumption from the last successful milestone.
- **FR-014**: System MUST support `--resume` flag to continue from saved state, skipping already-completed phases and services while re-loading their contracts for verification.
- **FR-015**: System MUST prevent concurrent orchestration runs on the same project via a lock mechanism, reporting an error if an existing run is detected.
- **FR-016**: System MUST display real-time progress showing the phase map, per-service status, verification results, and a final summary report.
- **FR-017**: System MUST generate an integration report upon completion (or failure) summarizing: services implemented per phase, verification results at each boundary, integration validation results, and overall status.
- **FR-018**: System MUST validate that all declared services have required spec artifacts (at minimum tasks.md) before beginning execution, reporting any missing artifacts as pre-flight errors.
- **FR-019**: System MUST skip services with missing spec artifacts (logging a warning) and also skip any services that depend on skipped services, with a clear explanation of the dependency chain.
- **FR-020**: When a service fails within a phase (beyond the sub-agent executor's retry budget), the system MUST allow remaining independent services in the same phase to continue (since same-phase services have no mutual dependencies), report the failure, and halt before proceeding to subsequent phases.
- **FR-021**: System MUST report all contract violations found during verification (not just the first one), enabling the developer to address multiple issues in a single fix cycle.
- **FR-022**: System MUST support `--to-phase N` flag with `--all` to execute only up to (and including) the specified phase number, enabling incremental development and testing of partial service sets.
- **FR-023**: System MUST auto-generate cross-service integration tests from published contracts (API specs, event schemas) during the post-phase integration validation, ensuring every contract between implemented services has at least one corresponding integration test.
- **FR-024**: System MUST use docker-compose (the same configuration generated by the shared infrastructure pre-phase) for integration validation — not an alternative container testing library — so that integration tests reflect actual deployment topology.
- **FR-025**: In microservice mode, each service MUST own its own database instance. Database migrations are scoped to the owning service and run as part of that service's implementation phase. No service may directly access another service's database.
- **FR-026**: In monolithic mode, modules share a single database but each module MUST own a distinct schema (or table namespace). Module boundary compliance checks (FR-010) must verify no cross-module direct database access outside the owning module's schema.

### Key Entities

- **Orchestration Plan**: The computed execution plan derived from the manifest's dependency graph. Contains the ordered list of phases, each phase containing a set of independent services/modules. Includes the pre-phase (shared infrastructure) and post-phase (integration validation) steps. Supports partial execution via a phase ceiling (`--to-phase N`).
- **Phase**: A group of services/modules that share the same dependency depth and can be implemented independently of each other. Phases are ordered such that all dependencies of a phase's services are satisfied by prior phases. If one service in a phase fails, the remaining services still complete.
- **Orchestration State**: The persistent record of orchestration progress. Tracks which phases are complete, which services within each phase are complete, verification results at each boundary, and the current execution position. Used for resumption after interruption.
- **Verification Result**: The outcome of inter-phase verification. Contains contract check results (per service pair), shared entity boundary analysis results, infrastructure health check results, and database schema ownership checks. Each result is pass/fail with diagnostic details on failure.
- **Integration Report**: The final output produced after all phases complete (or after failure). Summarizes the entire orchestration run: phase-by-phase results, service-by-service status, verification outcomes, auto-generated integration test results, and overall verdict.
- **Service Database**: In microservice mode, each service owns an isolated database instance with its own migrations. In monolith mode, modules own distinct schemas within a shared database. No cross-boundary direct database access is permitted in either mode.

## Assumptions

- Features 001–010 are complete and operational, providing: project manifest with architecture mode and service dependencies (Feature 004), per-service spec artifacts including tasks.md (Features 005–008), single-service sub-agent executor with quality gates and auto-fix (Features 009–010), shared infrastructure executor (Feature 009 US4), and shared entity boundary analysis (Feature 006).
- The sub-agent executor (Feature 009) handles all within-service complexity (task ordering, quality checks, auto-fix loops, commits). The orchestrator delegates to it and only concerns itself with cross-service coordination.
- Contract verification uses the contracts published by each service during implementation (API specs, event schemas). The mechanism for publishing and reading contracts is established by the shared infrastructure pre-phase and the sub-agent executor.
- In microservice mode, Docker and docker-compose are available on the developer's machine for integration validation. The orchestrator does not install or configure Docker itself.
- The manifest's dependency declarations are accurate and complete. The orchestrator trusts the manifest and does not infer dependencies from code analysis.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A developer can run a single command (`specforge implement --all`) and have all services in a multi-service project built in the correct order with zero manual intervention between phases.
- **SC-002**: Contract violations between services are detected at the phase boundary where they are introduced, not deferred to a final integration step — reducing the time between introducing and discovering cross-service bugs to the length of one phase.
- **SC-003**: 100% of dependency ordering errors are caught before execution begins (cycle detection, missing services) rather than discovered mid-run.
- **SC-004**: A monolithic project's `implement --all` experience is indistinguishable from "just works" — no container prompts, no contract verification noise, no microservice-specific output.
- **SC-005**: An interrupted orchestration run (after fixing any issues) resumes in under 30 seconds of re-analysis, regardless of how many services were already completed.
- **SC-006**: The integration report provides a single pass/fail verdict within 5 minutes of the last service completing, with per-service breakdown sufficient to diagnose any failures without additional investigation.
