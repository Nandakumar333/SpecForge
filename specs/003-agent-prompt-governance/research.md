## Topic 1: Structured Markdown Parsing (Zero External Deps)

**Decision**: Heading-based section traversal using Python `re` + `str.split()`

**Rationale**: Constitution Gate G-02 prohibits external dependencies in `core/`.
The governance file format uses a predictable structure: `## Meta`, `## Rules`,
`### {RULE-ID}: {Title}`, `threshold:` key-value lines. Python's built-in `re`
module is sufficient to extract all fields deterministically.

**Parsing strategy**:
1. Split file content on `\n### ` to get rule blocks
2. Within each rule block, extract fields via `re.match(r"^(\w+):\s*(.+)$", line)`
   for single-line fields (severity, scope, threshold)
3. Extract multi-line `example_correct`/`example_incorrect` as everything between
   the key line and the next key line (strip leading `  ` indent from YAML-style blocks)
4. Parse `## Meta` block as simple `key: value` lines

**Limitations accepted**: Parser is strict ŌĆö files not following the template
structure will return parse errors. This is intentional (governance files are
generated by SpecForge, not hand-authored in freeform Markdown).

**Alternatives considered**:
- `python-frontmatter`: External dep, prohibited in core/
- `mistune` Markdown AST: External dep, overkill for a known-structure format
- `tomllib` (built-in Python 3.11): TOML is not Markdown; would require changing
  the governance file format, violating the user-editable Markdown requirement

## Topic 2: File Naming Convention (`domain.stack.prompts.md`)

**Decision**: `{domain}.{stack}.prompts.md` for stack-specific files,
`{domain}.prompts.md` for agnostic domains (architecture, security)

**Rationale**: Allows multiple stack variants to coexist in `.specforge/prompts/`
without collision. Consistent with the existing template naming convention in
`TemplateRegistry` (e.g., `backend.dotnet.md.j2`). `PromptLoader` uses a
2-step resolution: try stack-specific filename first, fall back to generic.

**Resolution chain** (implemented in `PromptLoader._resolve_path`):
```text
1. .specforge/prompts/{domain}.{stack}.prompts.md
2. .specforge/prompts/{domain}.prompts.md
3. ŌåÆ Err if neither found
```

**Agnostic domains** (`architecture`, `security`) always use the generic filename
regardless of stack, because their content is intentionally stack-independent.

**Alternatives considered**:
- Flat `{domain}.prompts.md` with stack metadata inside ŌĆö prevents side-by-side
  multi-stack use; a future dotnet+nodejs monorepo would need to overwrite files
- Single file per project, dynamically selected at load time ŌĆö confuses `--force`
  customization detection

## Topic 3: SHA-256 Checksum for Change Detection (`--force` behavior)

**Decision**: Embed SHA-256 hash of original rendered content in `## Meta` section
as `checksum: {hex}`. On `--force`, re-render and compare; skip regeneration
if hashes differ (file was customized).

**Implementation** (`hashlib.sha256` ŌĆö stdlib, no external deps):
```python
import hashlib

def _compute_checksum(content: str) -> str:
    return hashlib.sha256(content.encode()).hexdigest()
```

**Checksum lifecycle**:
1. `generate()` renders template ŌåÆ strips any existing checksum line ŌåÆ computes hash
   of clean rendered text ŌåÆ injects `checksum: {hex}` into `## Meta` block
2. `is_customized()` reads stored checksum, re-renders template, computes fresh hash,
   returns `stored != fresh`
3. If user edits any character in the file, the hash changes ŌåÆ `is_customized()` ŌåÆ True
   ŌåÆ `--force` preserves the file

**Edge case**: If the template itself changes (SpecForge upgrade), stored hash
and fresh hash diverge even for unedited files ŌåÆ correctly flags as "possibly
customized" ŌåÆ conservative behavior (preserves user's file). Acceptable trade-off.

**Alternatives considered**:
- External `.specforge/prompts/.checksums.json` tracking file ŌĆö requires separate
  file management, another failure mode (out of sync), harder to explain to users
- Git-based change detection (`git diff`) ŌĆö requires git to be initialized and
  all files committed; breaks `--no-git` projects

## Topic 4: `TemplateType.governance` vs Reusing `TemplateType.prompt`

**Decision**: Add `TemplateType.governance` to the enum in `template_models.py`
and scan `templates/governance/` subdirectory in `TemplateRegistry._discover_built_in()`.

**Rationale**: The existing `TemplateType.prompt` covers agent-operation prompts
(app-analyzer, feature-specifier, etc.) ŌĆö files that tell agents *how to work*.
Governance templates tell agents *what rules to enforce*. These are semantically
distinct and appear in different output locations (`.specforge/prompts/` for
governance vs legacy paths for agent-operation prompts). Keeping them separate
prevents `scaffold_builder.py` from accidentally mixing them.

**Registry extension**: Add `"governance": TemplateType.governance` to `_TYPE_MAP`
in `template_registry.py`. The existing `_scan_directory` + `_parse_template_file`
logic handles the new directory automatically (reuses `domain.stack.md.j2` parsing).

**Alternatives considered**:
- Reusing `TemplateType.prompt` with a name prefix (e.g., `gov-backend`) ŌĆö
  fragile naming convention, `scaffold_builder.py` can't distinguish them cleanly

## Topic 5: Project Stack Persistence (`.specforge/config.json`)

**Decision**: Write `.specforge/config.json` with `{stack, project_name, created_at}`
during `specforge init`. `PromptLoader` reads this file to determine which stack's
governance files to load.

**Rationale**: `PromptLoader.load_for_feature(feature_id: str)` takes no stack
parameter. The loader must independently resolve which files to load. A small JSON
config file is the simplest persistent store ŌĆö stdlib `json` module, zero deps.

**Schema** (minimal, forward-extensible):
```json
{
  "project_name": "myapp",
  "stack": "dotnet",
  "specforge_version": "0.1.0",
  "created_at": "2026-03-15"
}
```

**`StackDetector` integration** (FR-016): When `--stack` is omitted from
`specforge init`, auto-detect stack before writing `config.json`:
```text
Markers scanned (first match wins):
  *.csproj       ŌåÆ dotnet
  package.json   ŌåÆ nodejs
  pyproject.toml | requirements.txt ŌåÆ python
  go.mod         ŌåÆ go
  pom.xml | build.gradle ŌåÆ java
  (no markers)   ŌåÆ agnostic
```

**Alternatives considered**:
- Infer stack from filename patterns in `.specforge/prompts/` at load time ŌĆö
  fragile, breaks with agnostic files (architecture.prompts.md, security.prompts.md)
- Store in `.specforge/constitution.md` YAML frontmatter ŌĆö constitution is a
  Markdown doc for humans; mixing machine-readable config into it couples concerns

## Topic 6: Conflict Detection Algorithm

**Decision**: Single-pass threshold index with O(n) complexity where n = total
threshold entries across all 7 files.

**Conflict identification**:
- Index: `threshold_key ŌåÆ [(domain, rule_id, value), ...]`
- A conflict exists when the same `threshold_key` appears with Ōēź2 distinct values
- Conflict type:
  - **Cross-priority** (e.g., `architecture` vs `backend`): winner = higher-precedence
    domain; loser should update its value to match winner
  - **Intra-priority** (e.g., `backend` vs `database`): `is_ambiguous=True`;
    no winner declared; both files shown in report with manual resolution instruction

**Threshold key uniqueness**: Rule authors MUST use globally unique threshold keys
across all files (e.g., `max_class_lines` should appear in only one file, or
intentionally duplicated if the same constraint applies domain-wide). The validator
reports all duplicates ŌĆö even intentional duplicates where values agree ŌĆö as informational.
Actual conflicts only reported when values differ.

**Alternatives considered**:
- Rule ID matching (conflict if same `SOLID-001` in two files) ŌĆö rule IDs are
  domain-scoped by design (backend SOLID-001 ŌēĀ architecture SOLID-001); threshold
  keys are the right comparison unit for numeric conflict detection

## Topic 7: Performance Guarantee (Ōēż500 ms)

**Decision**: Direct file I/O + regex parsing with no caching layer.

**Justification**: 7 Markdown files ├Ś ~300 lines each = ~2,100 lines total.
Python's file I/O reads this in <5 ms. Regex parsing of structured text adds
<10 ms. Total wall time is dominated by process startup, not file reading.
No caching required for the Ōēż500 ms SLA.

**Benchmark reference**: `pathlib.Path.read_text()` on a 500-line file on
a spinning HDD completes in ~1ŌĆō3 ms. SSD is sub-millisecond. The 500 ms budget
provides >100├Ś headroom ŌĆö no optimization needed.
